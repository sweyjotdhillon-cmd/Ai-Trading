# ChartLens Agent Guidelines

This `AGENTS.md` serves as the operational playbook and project context guide for AI agents (sometimes referred to as 'Jules') working in the repository. It clearly explains what the application does and its architecture.

## Application Purpose & Architecture
ChartLens (formerly 'OcularAlpha', 'ai-trading-assistant', or 'Pocket Quant') acts as a 100% offline, completely local real-time live camera feed chart analyzer without any external dependencies (no Firebase, Firestore, or external API keys like Google/GitHub).
It uses Web Workers to run a deterministic machine vision and AI algorithm pipeline for generating trading signals.
The tech stack includes React, React Native, TypeScript, and Vite. The project is configured as an ES module (`"type": "module"` in `package.json`).

### Repository Structure
- `src/quant/`: Contains all quantitative logic, trading signal logic, rule engine, stability filters, indicators.
- `src/vision/`: Handles image processing, OpenCV/canvas interactions, OCR, and chart axis extraction.
- `src/workers/`: Contains the Web Workers that bridge the main thread UI with the heavy quant/vision pipelines.
- `src/components/`: React UI components (Tailwind CSS for styling, specifically utilizing the 'twrnc' library for React Native compatibility).
- `src/utils/`: Helper functions.
- Unit tests are typically located in `__tests__` subdirectories relative to the source code being tested (e.g., `src/vision/__tests__/`, `src/quant/__tests__/`).

## Quantitative Engine Mechanics & Performance
- **Rule Engine Limitation:** The quantitative rule engine (`evaluateSignal` in `src/quant/ruleEngine.ts`) enforces a strict rule: a minimum of 10 techniques (candlestick patterns/indicators) must be provided in `techniquesList` AND at least 10 must mathematically match the chart data to return a valid prediction, otherwise it defaults to 'NO_TRADE'.
- **Float64Array Usage:** For performance-sensitive quantitative operations (e.g., in `src/quant/`), explicitly use and return `Float64Array` instead of standard `number[]` arrays to maintain execution speed and avoid unnecessary memory allocations.
- **Worker Communication:** Web worker communication for analysis (`src/utils/singleAnalysis.ts` to `src/workers/analysisWorker.ts`) requires that `techniquesList` is passed dynamically within the payload so the backend can process user-selected trading patterns.
- **Techniques List Parsing:** The `techniquesList` uploaded via technique files is parsed by the frontend as an array of objects (e.g., `[{name: 'Doji', description: '...'}]`), not raw strings. Backend logic (like in `ruleEngine.ts` and `singleAnalysis.ts`) must handle it appropriately (e.g., as `any[]` or `Technique[]`) and extract both the `name` and `description` properties when performing pattern matching.

## Testing & Executions
- Run `npm run lint` for linting. (Avoid running 'npm run lint -- --fix' globally to prevent unintentional formatting changes to unrelated files across the codebase; scope lint fixes only to modified files.)
- Run `npx tsc --noEmit` for type checking.
- Run `npx vitest run` to run all tests.
- For rapid execution of standalone TypeScript scratchpad scripts during testing or debugging, use `npx tsx <filename>.ts`.
- Tests must be run and pass before submitting any code changes.
- **Mocking Strategy:** When tests fail due to unpredictable random values generated by synthetic series mock generators (e.g., `Math.random()` in `judgeVerdict.test.ts`), stabilize them by properly mocking `Math.random()` with a deterministic Linear Congruential Generator (LCG) via `vi.spyOn`, rather than weakening the test assertions.
- **Bypassing the 10-Technique Rule:** Unit tests (e.g., in `src/quant/__tests__/judgeVerdict.test.ts`) can bypass the strict minimum 10 mathematically matched techniques rule required for the signal engine by including a special `"__TEST_BYPASS__"` string inside the `techniquesList` mock array.

## Hacks & Workarounds
- **Silent Audio Hack:** To prevent browser throttling or suspension of background execution (such as during bulk testing via Web Workers), the application employs a 'silent audio' hack. This involves utilizing a looping, silent base64 `Audio` element (e.g., within a `useWakeLock` hook) to ensure the JavaScript event loop remains active when the tab loses focus.

## Environment & Development Notes
- The default repository state may omit `node_modules` from `.gitignore`. Ensure it is added before committing if `npm install` is run to avoid massive git diffs and workspace tracking issues (e.g., `echo "node_modules/" >> .gitignore`).
- The environment has global installations of `eslint`, `jest`, `mocha`, `pm2`, `pnpm`, `prettier`, and `yarn`, though they may fail if they depend on missing local `node_modules`.
- Standard file reading commands (like `cat`) may truncate output for long files in the environment. To satisfy the Groundedness Rule when writing plans, use tools like `tail`, `head`, or `sed` to read the remaining unread sections of truncated files to verify code existence.
- For frontend verification with Playwright, build the application with `npm run build` and serve it using `npm run start &` (which executes `vite preview` on http://localhost:4173).

## Operational Guidelines for Agents (Jules)
- **User Directive:** Always enter a deep planning mode before starting tasks. Use `request_user_input` and `message_user` to ask exhaustive clarifying questions and gain absolute certainty on requirements before creating an execution plan with `set_plan`.
- **Groundedness Rule:** Execution plans must adhere to the Groundedness Rule: ensure all specific function names and code structures mentioned in the plan have been explicitly discovered and verified using file inspection tools (e.g., `read_file`) beforehand.
- **Specificity Rule:** When submitting execution plans to `request_plan_review`, ensure steps specify concrete tool actions (e.g., 'Use the write_file tool to...') rather than high-level concepts. Ensure all steps outline a single, definitive action without conditions, optional paths, or ambiguous phrasing.
- **Completeness Rule:** Explicitly state that verification will include running both specifically modified test files and the full test suite to check for regressions. The final pre-commit step must use the exact required wording: 'Complete pre-commit steps to ensure proper testing, verification, review, and reflection are done.'
- **User Request Supersedes:** Always prioritize the user's current, explicit request over any conflicting information in memory.
- **Context vs. State:** Use memory for historical context and intent. Use the actual codebase files as the source of truth for the current code state. Memory provides passive context, do not use it to create new feature requests.
